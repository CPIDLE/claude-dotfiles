# ADR 001: Transition from Opencode CLI to LangGraph-orchestrated Gemini Pipeline

## Status
Proposed

## Context
Our current CLI task delegation system relies on a rigid piping mechanism where Claude Code generates specifications that are executed by the `opencode` CLI. While this fulfills basic automation needs, it suffers from several architectural constraints:
* **Linearity:** The process is strictly unidirectional; there is no native support for feedback loops or multi-turn reasoning.
* **Statelessness:** Session context is lost upon process termination, preventing long-running task resolution.
* **Opaque Logic:** The `opencode` CLI operates as a black box, making it difficult to inject custom conditional routing, safety checks, or specialized error-handling logic.

To improve reliability and extensibility, we need a system that supports stateful workflows, branching logic, and fine-grained control over model interaction.

## Decision
We will migrate the execution architecture to a LangGraph-based workflow. This system will interact directly with the Gemini API using native SDKs rather than wrapping the `opencode` CLI. Key components include:
* **Orchestration:** A directed graph using LangGraph to manage nodes for `Classify`, `Execute`, `Validate`, and `Report`.
* **Persistence:** Use of SQLite (via `SqliteSaver`) to maintain state across execution steps and sessions.
* **Control Flow:** Implementation of conditional edges to handle task retries, human-in-the-loop (HITL) approval, and error recovery transitions.
* **Direct Integration:** Replacing the `opencode` CLI dependency with direct Gemini API calls to reduce latency and gain granular access to model configuration (e.g., temperature settings, tool calling schemas).

## Consequences

### Positive
* **Statefulness:** SQLite persistence allows for resuming interrupted tasks and provides an audit trail of agentic reasoning.
* **Extensibility:** New nodes (e.g., security scanning, code linting) can be injected into the graph without altering the core execution loop.
* **Observability:** Granular visibility into the transition between graph nodes via LangGraph’s built-in tracing capabilities.
* **Resilience:** The ability to define retry cycles and fallback paths significantly improves success rates for complex tasks.

### Negative
* **Increased Complexity:** Moving from a simple CLI pipe to a managed state machine increases the maintenance overhead of the codebase.
* **Infrastructure Dependency:** We transition from a binary dependency (`opencode`) to a library-heavy dependency (LangGraph/Pydantic/Google GenAI SDK).
* **Initial Migration Effort:** The current spec format generated by Claude Code may require transformation to align with the new graph’s input requirements.

## Alternatives Considered
* **Retaining Opencode CLI with External State:** Attempting to wrap `opencode` in a script that manages SQLite states. Rejected because it fails to address the "black box" limitation and the lack of complex branching control.
* **Migrating to LangChain Chains:** Using `SequentialChain` instead of LangGraph. Rejected because simple chains lack the cyclical dependencies and robust state management required for complex task delegation and error recovery.
* **Custom Python Orchestrator:** Building a state machine from scratch using Python `asyncio`. Rejected due to the high development cost and maintenance burden compared to the mature primitives provided by LangGraph.
