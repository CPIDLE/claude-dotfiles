# RT-Fact — 事實基礎層紅隊

針對「cc 衍生檔污染」的輕量化檢查。三條規則，目標 5–15 分鐘完成。

> **背景**：cc 整理的 `.md` / review / 統整筆記一旦進入引用鏈，framing 污染會跨會話擴散。本指令只攻**事實基礎層**（引文、數量、措辭），不攻論點層。論點層走 `/pm-review`。
>
> 設計來源：`docs/redteam-proposal.md`（ABL 案、架構整理污染案）

## 觸發

- 手動：`/rt-fact <檔案路徑>` 或 `/rt-fact`（預設審當前 git diff + 最近修改的 `.md`）
- 自動：`/pm-review` Step 0 偵測到引用 cc 衍生檔時自動呼叫

## 範圍判定

**白名單（原始來源）**：`*.docx` / `*.pdf` / `*.png` 截圖 / 微信逐字稿 / 公開公告 / user 直接打字 / 程式碼原始檔

**黑名單（cc 衍生檔，需審）**：`reviews/`、`*review*`、`*-summary*`、`*整理*`、`*筆記*`、`*統整*`、`*analysis-v*`、`*vN.md`、對話 compaction 結果

## 三條規則（平行執行）

### Rule 1｜來源鏈追溯

對被審文件中**每個關鍵引文 / 事實宣稱**：

1. 標註聲稱的來源
2. 用 Grep 在白名單檔案中找逐字對應
3. 若來源是另一份 cc 衍生檔 → **繼續溯源**直到原始檔
4. 找不到原始對應 → 標 `❌ cc 自創` 或 `⚠️ 無法追溯`

### Rule 2｜定性 ↔ 數量對質

對所有形容詞性質判斷（「入主」「不可行」「偏高」「大概率」「主要」「構成 / 不構成」⋯）：

1. 對應的數量是什麼？（持股 % / 倍數 / 金額 / 條件數量 / 時間）
2. 數量在原文哪裡？（行號 / 段落）
3. 數量是否支持該定性？

任一空 / 矛盾 → 定性站不住，標 `❌ 矛盾` 或 `⚠️ 邊界模糊`

### Rule 3｜灰色詞 grep（30 秒）

對被審文件 grep：

```
類|準|擬|原|部分|一定|可能|暫|初步|大致|某種|相對|據說|似乎
```

每個命中位置：

- 原文也有 → ✅ OK
- 原文有、被審文件**省略限定詞** → ❌ framing 污染（最常見）
- 被審文件加了原文沒有的限定詞 → ⚠️ cc 軟化原意

## 輸出格式

寫入 `reviews/YYYY-MM-DD-HH-MM-rtfact.md`：

```markdown
# RT-Fact: <檔名>
日期: YYYY-MM-DD HH:MM
被審範圍: <檔案清單>
追溯到的原始來源: <清單>

## Rule 1 來源鏈
| 引文 / 宣稱 | 聲稱來源 | 原始對應 | 裁決 |
|---|---|---|---|
| ... | ... | ... | ✅ / ⚠️ / ❌ |

## Rule 2 數量對質
| 定性 | 對應數量 | 原文位置 | 支持？ |
|---|---|---|---|
| ... | ... | ... | ✅ / ⚠️ / ❌ |

## Rule 3 灰色詞
| 詞 | 被審位置 | 原文對應 | 裁決 |
|---|---|---|---|
| ... | ... | ... | ✅ / ❌ 省略 / ⚠️ 軟化 |

## 整體結論
✅ PASS / ⚠️ 部分污染（列段落）/ ❌ 全篇打回（列原因）
```

## Step 1：螢幕摘要

報告寫完後，輸出一行：

```
🧪 RT-Fact: <✅ PASS / ⚠️ 部分污染 / ❌ 全篇打回> | R1: N issues | R2: N issues | R3: N issues
📋 reviews/YYYY-MM-DD-HH-MM-rtfact.md
```

❌ 全篇打回 / ⚠️ 多處污染 → 暫停，詢問是否修正後再繼續下游分析。

## 邊界

- **不攻論點**（條件鏈、selection bias、隱含假設）→ 走 `/pm-review`
- **不做鏡像 read**（從零重寫對照）→ 成本太高，改用人工抽查
- **黑/白名單有灰色地帶**（如 user 轉述 / OCR 加註）→ 依 `docs/redteam-proposal.md` 第三節判定表
