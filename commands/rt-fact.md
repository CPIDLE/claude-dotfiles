# RT-Fact — 事實基礎層紅隊

針對「cc 衍生檔污染」的輕量化檢查。三條規則，目標 5–15 分鐘完成。

> **背景**：cc 整理的 `.md` / review / 統整筆記一旦進入引用鏈，framing 污染會跨會話擴散。本指令只攻**事實基礎層**（引文、數量、措辭），不攻論點層。論點層走 `/pm-review`。
>
> 設計來源：`docs/redteam-proposal.md`（ABL 案、架構整理污染案）

## 模式

| 模式 | 觸發 | 執行者 | 時間 | 產出 |
|---|---|---|---|---|
| **lite**（預設） | `/rt-fact <檔>`、`/pm-review` Step 0 自動 | 主 agent | 5–15 分鐘 | `reviews/YYYY-MM-DD-HH-MM-rtfact.md` |
| **deep** | `/rt-fact --deep <檔>`、lite 命中 ⚠️/❌ 後使用者升級 | 獨立 subagent | 30–60 分鐘 | `reviews/YYYY-MM-DD-HH-MM-rtfact-deep.md` |

> deep 多一層**鏡像 read**：禁讀 cc 衍生檔，從零重寫對照。用來攻 lite 攔不到的「同一個 cc 被自己 framing 污染」盲點。

## 範圍判定

**白名單（原始來源）**：`*.docx` / `*.pdf` / `*.png` 截圖 / 微信逐字稿 / 公開公告 / user 直接打字 / 程式碼原始檔

**黑名單（cc 衍生檔，需審）**：`reviews/`、`*review*`、`*-summary*`、`*整理*`、`*筆記*`、`*統整*`、`*analysis-v*`、`*vN.md`、對話 compaction 結果

## 三條規則（平行執行）

### Rule 1｜來源鏈追溯

對被審文件中**每個關鍵引文 / 事實宣稱**：

1. 標註聲稱的來源
2. 用 Grep 在白名單檔案中找逐字對應
3. 若來源是另一份 cc 衍生檔 → **繼續溯源**直到原始檔
4. 找不到原始對應 → 標 `❌ cc 自創` 或 `⚠️ 無法追溯`

### Rule 2｜定性 ↔ 數量對質

對所有形容詞性質判斷（「入主」「不可行」「偏高」「大概率」「主要」「構成 / 不構成」⋯）：

1. 對應的數量是什麼？（持股 % / 倍數 / 金額 / 條件數量 / 時間）
2. 數量在原文哪裡？（行號 / 段落）
3. 數量是否支持該定性？

任一空 / 矛盾 → 定性站不住，標 `❌ 矛盾` 或 `⚠️ 邊界模糊`

### Rule 3｜灰色詞 grep（30 秒）

對被審文件 grep：

```
類|準|擬|原|部分|一定|可能|暫|初步|大致|某種|相對|據說|似乎
```

每個命中位置：

- 原文也有 → ✅ OK
- 原文有、被審文件**省略限定詞** → ❌ framing 污染（最常見）
- 被審文件加了原文沒有的限定詞 → ⚠️ cc 軟化原意

## 輸出格式

寫入 `reviews/YYYY-MM-DD-HH-MM-rtfact.md`：

```markdown
# RT-Fact: <檔名>
日期: YYYY-MM-DD HH:MM
被審範圍: <檔案清單>
追溯到的原始來源: <清單>

## Rule 1 來源鏈
| 引文 / 宣稱 | 聲稱來源 | 原始對應 | 裁決 |
|---|---|---|---|
| ... | ... | ... | ✅ / ⚠️ / ❌ |

## Rule 2 數量對質
| 定性 | 對應數量 | 原文位置 | 支持？ |
|---|---|---|---|
| ... | ... | ... | ✅ / ⚠️ / ❌ |

## Rule 3 灰色詞
| 詞 | 被審位置 | 原文對應 | 裁決 |
|---|---|---|---|
| ... | ... | ... | ✅ / ❌ 省略 / ⚠️ 軟化 |

## 整體結論
✅ PASS / ⚠️ 部分污染（列段落）/ ❌ 全篇打回（列原因）
```

## Step 1：螢幕摘要

報告寫完後，輸出一行：

```
🧪 RT-Fact: <✅ PASS / ⚠️ 部分污染 / ❌ 全篇打回> | R1: N issues | R2: N issues | R3: N issues
📋 reviews/YYYY-MM-DD-HH-MM-rtfact.md
```

❌ 全篇打回 / ⚠️ 多處污染 → 暫停，詢問是否修正後再繼續下游分析。

## Deep 模式

啟動**獨立 subagent**，鏡像重做事實基礎層分析，與被審文件 diff 找出隱性污染。

### 升級提示

兩種觸發升級的訊號（任一命中即提示）：

1. **lite 結果為 ⚠️ / ❌**：
   ```
   💡 lite 發現潛在污染。要升級 deep 鏡像 read？(/rt-fact --deep <檔>)
   ```

2. **被審議題涉及高風險訊號**（grep 命中以下任一詞）：
   ```
   重大|不可逆|鉅額|策略性|定性|構成|不構成|對外承諾|正式簽|法律責任|第三方|對外公告|對外披露|客戶承諾|供應商|合作方|合資|投資人
   ```
   ```
   💡 議題涉及高風險訊號（<列命中詞>），強烈建議升級 deep。(/rt-fact --deep <檔>)
   ```

兩條件同時命中 → 合併為一條更強的提示：「lite 已見污染 + 高風險議題 → **務必**升級 deep」。

> deep 不自動執行（30–60 分鐘成本），由使用者決定。

### Subagent prompt

```
你是獨立事實基礎層紅隊。立場：被審文件可能被自己的整理者（cc）framing 污染。

## 硬性禁令
- 禁讀任何 cc 衍生檔（reviews/、*review*、*整理*、*筆記*、*統整*、*summary*、*analysis-v*、vN.md）
- 禁讀對話 compaction 結果
- 禁讀被審文件本身（直到鏡像分析完成）

## 允許讀的來源（白名單）
- *.docx / *.pdf 原始方案文件
- screenshot 圖片 / OCR（cc 未改動）
- 微信原始對話 / 會議逐字稿
- 公開公告 / 年報 / 新聞
- 程式碼原始檔（git tracked）
- user 直接打字提供的事實

## 任務
1. **鏡像 read**：只用白名單來源，從零寫一份 < 500 字事實摘要，涵蓋被審議題的關鍵事實、數量、措辭
2. **diff 對照**：讀取被審文件，逐段與你的鏡像分析比對：
   - 事實一致 → ✅
   - 數量 / 措辭差異 → ⚠️ 列出兩邊原文
   - 結論方向相反 → ❌ 列出兩邊原文
3. **污染類型分類**：
   - framing 污染（限定詞被省略 / 加重）
   - 動詞污染（cc 自創措辭，原文沒有）
   - 數量遺漏（定性結論取代量化）
   - 來源錯置（引用另一份 cc 衍生檔當原始來源）
   - 斷章（條款 / 上下文被抽離）

## 報告
寫入 reviews/YYYY-MM-DD-HH-MM-rtfact-deep.md：

# RT-Fact Deep: <檔名>
日期: YYYY-MM-DD HH:MM
被審文件: <路徑>
白名單來源: <清單>

## 鏡像分析（< 500 字）
<從零重寫的事實摘要>

## Diff 對照
| 段落 | 被審文件 | 鏡像分析 | 污染類型 | 嚴重度 |
|---|---|---|---|---|
| ... | ... | ... | framing/動詞/數量/來源/斷章 | 🔴/🟠/🟡 |

## 污染證據鏈
<列出每一處污染的最短復現路徑：原文→cc 整理→下游引用>

## 整體結論
✅ 鏡像一致 / ⚠️ N 處污染需修正 / ❌ 全篇基礎不可信
```

附上被審文件路徑 + 白名單來源清單作為素材。

### Step 3：螢幕摘要

subagent 完成後輸出：
```
🧪 RT-Fact Deep: <✅ 一致 / ⚠️ N 處污染 / ❌ 全篇打回>
📋 reviews/YYYY-MM-DD-HH-MM-rtfact-deep.md
```

❌ 全篇打回 → 暫停，詢問是否打回重做。

## 邊界

- **不攻論點**（條件鏈、selection bias、隱含假設）→ 走 `/pm-review`
- **無 `--auto-deep`**：deep 成本高，避免無預警吃 token；要升級必須手動
- **黑/白名單有灰色地帶**（如 user 轉述 / OCR 加註）→ 依 `docs/redteam-proposal.md` 第三節判定表
