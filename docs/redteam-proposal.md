# Red-team 流程修訂建議

> 日期：2026-05-03
> 觸發來源：投資討論_V0 / ABL 案 5/3 對抗分析
> 已發生案例：cc 連續 2 週、6+ 份衍生材料、3 輪原 RT 都沒攔到 framing 污染（「Gyro 入主」是 cc 自創、原 docx 寫第二大股東）；最終靠 user 一句「只引原始文件」才暴露。

---

## 一、問題診斷

現行 RT **攻擊論點強度，但接受事實基礎為真**。看不到的盲點：

| 層 | 現行 RT |
|---|---|
| 事實基礎層（引文、原文措辭、來源鏈） | ❌ 假設為真 |
| 論點層（推論、條件鏈、解讀） | ✅ 主戰場 |

最常見污染：
1. **「類 X / 準 X / 擬 X」灰色詞被簡化** — 「類借殼」→「借殼」
2. **動詞污染** — 原 docx 沒有「接手」「入主」，cc 整理時自創
3. **跳過數量參數** — 原 docx「第二大股東」「11%」這類數量結論被「角色定位」式形容詞取代
4. **「原 X」斷章取義** — 條款語境（現任繼續）被抽出做標籤式描述（前任）
5. **Framing 與引用未分離** — cc 用 `>` 引用塊格式寫自己的詮釋，下游引用者無法分辨

擴散路徑：cc 整理材料一旦寫入下游文件就成為 second-source，後續引用無法還原 framing 之前的原意。

---

## 二、修訂方案：三層 RT 架構

### Layer 1: RT-lite（事實基礎層，10–15 分鐘）

對應現行 RT 失手的污染。**三條規則**：

#### Rule 1｜每個關鍵引文必有「一條鏈到原始來源」
- 標註 `[來源: 檔名 L行號]`
- **禁止**：來源為另一份 cc 整理檔（`.md`、`reviewVx`、統整報告） → 必須繼續溯源
- 速查法：grep 該引文是否能在原始檔案找到逐字對應

#### Rule 2｜每個定性結論要對到一個數量
對任何形容詞性質的判斷（「入主」「不可行」「偏高」「大概率」「不構成」⋯）問三句：
1. 對應的數量是什麼？（持股比例 / 倍數 / 金額 / 百分比 / 條件數量）
2. 數量在原文哪裡？
3. 數量是否支持定性？

任一空 / 矛盾 → 定性站不住。

#### Rule 3｜灰色詞 grep（30 秒）
對被審文件 grep：`類|準|擬|原|部分|一定|可能|暫|初步`

每個命中位置問：**這個字在原文也有嗎？**
- 原文有、被審文件也有 → OK
- 原文有、被審文件**省略** → framing 污染（最常見）
- 被審文件有、原文沒有 → cc 自加（次常見）

### Layer 2: RT-original（論點層，30–60 分鐘）

現行 RT 方法。在 Layer 1 通過後執行。攻擊：
- 條件鏈跳躍
- Selection bias
- 解讀過度延伸
- 反向錨點遺漏
- 隱含假設

### Layer 3: RT-deep（鏡像 read，60+ 分鐘）

**禁用對 cc 整理檔的 read/grep**，只能讀：
- 原始 docx / PDF / screenshot OCR
- 原始微信對話、會議逐字稿
- 公開資料（年報、新聞、公告）

從零做一份極簡分析（< 500 字），與主分析做 diff。不一致 → 主分析有污染，重做 Layer 1。

## 三、觸發規則（依來源類型）

| 來源類型 | Rule 1 | Rule 2 | Rule 3 | 後續層級 |
|---|---|---|---|---|
| **純原始資料整理** | ❌ 跳 | ❌ 跳 | ✅ 跑（30 秒）| 原 RT |
| **二次整理 / 含衍生材料** | ✅ 跑 | ✅ 跑 | ✅ 跑 | 原 RT |
| **法律/結構/控制權定性議題** | 升級 deep（不論來源類型）| | | deep |

### 來源類型定義

**原始資料（白名單）**：
- `*.docx` / `*.pdf` 原始方案文件
- screenshot 圖片（含未經 cc 改動的 OCR）
- 微信原始對話 / 會議錄音逐字稿
- 公開公告、年報、新聞
- user 直接打字提供的事實

**二次整理（黑名單）**：
- cc 寫的盡調筆記、整理筆記
- cc 寫的 review、reviewVx
- cc 寫的統整報告、總結報告
- cc 寫的分析檔（含 vN 各版本）
- 對話 summary / compaction 結果
- 引用以上來源的任何衍生物

### 灰色地帶判定

| 情境 | 判定 |
|---|---|
| OCR 文字（cc 沒改動） | **原始**（內容未變形） |
| OCR 文字（cc 加了註解或選擇性引用） | **二次** |
| user 轉述 ST 講的話 | **原始**（user 是傳話而非整理者）|
| user 引用 cc 之前寫的東西讓 cc 評估 | **二次** |
| 純原始資料整理但 cc 寫了 framing 章節（角色定位、合作目標、結論） | 來源是原始，但 framing 章節仍要 Rule 3 |

## 四、自動觸發機制（建議實作）

```
on /redteam <檔案>:
    引用清單 = 解析該檔案所有引用
    if 任一引用 ∈ 黑名單:
        → 完整 LITE（三條）+ 原 RT
    elif 全部引用 ∈ 白名單:
        → 僅 Rule 3 灰色詞掃描 + 原 RT
    if 議題涉及「定性 / 結構 / 控制權 / 實控 / 借殼 / 重組上市 / 構成 / 不構成」:
        → 升級 deep（不論來源類型）
```

或拆三個 slash command：
- `/redteam` 預設（依來源類型自動判定）
- `/redteam --lite` 強制只跑三條
- `/redteam --deep` 強制完整版

## 五、輸出格式

### LITE 輸出（單頁）
```markdown
# RT-lite: [檔名]
日期: __

## Rule 1 來源鏈
- [關鍵引文 1] → 來源: __ → ✅ / ⚠️ 衍生材料 / ❌ cc 自創
- ...

## Rule 2 數量對質
- [定性 1]「__」→ 數量: __ → ✅ 支持 / ⚠️ 邊界模糊 / ❌ 矛盾
- ...

## Rule 3 灰色詞
被審文件 grep 結果: __
原文對應 diff: __

## 整體結論
通過 / 重做某段 / 全篇打回
```

### Deep 輸出（五段）
```markdown
# RT-deep: [檔名]

## L0 灰色詞掃描
## L1 來源鏈審計
## L2 數量參數對質
## L3 論點層攻擊
## L4 鏡像 read
## 整體結論
```

## 七、預期效益

| 情境 | 修訂前 | 修訂後 |
|---|---|---|
| 整理微信對話逐字稿 | 10–15 分鐘 LITE | 30 秒灰色詞掃描 |
| 寫盡調筆記（讀原 docx） | 10–15 分鐘 LITE | 30 秒灰色詞掃描 |
| 寫統整報告（引盡調筆記） | 10–15 分鐘 LITE | 完整 LITE |
| 寫對抗分析（引多份 cc 整理） | 10–15 分鐘 LITE | 完整 LITE |
| 法律定性議題 | 10–15 分鐘 LITE | 升級 deep（2–3 小時，但成本千萬級議題報酬率極高） |

降低 70% 不必要 RT 成本；保護高風險場景。

## 八、本提案的限制

1. **Rule 3 灰色詞名單是經驗性**，不窮盡（例如「某種」「相對」也是限定詞）
2. **數量參數對質**對純定性議題（如「治理風險」「商譽損益」）覆蓋不全
3. **L4 鏡像 read** 仍依賴同一個 cc 執行，無法完全跳出 framing pre-conception；理想是 RT 角色解耦（新 session）但實務成本高
4. **黑/白名單判定邊界**有灰色地帶，需依 case 判斷
5. ABL 案是 cc 自己 framing 污染；如果污染源是**外部材料**（張宇方提的不實主張），這套規則不直接覆蓋，需另一套 adversarial source check

## 九、實作優先級建議

1. **第一階段（必做）**：寫 `commands/redteam.md`，把三條 LITE 規則與觸發機制落地為 slash command
2. **第二階段（強烈建議）**：升級 reviews 流程，分 lite / deep 兩類輸出
3. **第三階段（可選）**：寫 skill 自動化整合 hook
4. **第四階段（理想）**：RT 角色解耦（新 session 機制）
