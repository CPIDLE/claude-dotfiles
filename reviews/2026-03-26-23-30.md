## 發現清單

### 🔴 阻塞 (MUST FIX)

---

**F-001**
- **位置**：計畫第 1 點（Status Line 指令分散且重複 — 砍掉改用 hook 處理）
- **嚴重度**：🔴
- **問題**：settings.json 的 hook 系統不支援「每個子命令開始/結束時自動觸發指定 shell 腳本」的能力
- **驗證**：交叉比對 `settings.json` 現有 hook 結構。目前只有 `PreToolUse`（匹配特定 tool name 觸發）和 `Stop`（session 結束觸發）兩種 hook。沒有 `PreCommand`、`PostCommand`、`OnSkillStart`、`OnSkillEnd` 或類似機制。slash command（如 `/pm sync`）是 skill markdown 檔案，不是 tool call，因此 `PreToolUse` matcher 無法攔截。要實現「/pm sync 開始時自動執行 pm-update.sh sync running」需要一個目前不存在的 hook type。
- **裁決**：**MUST FIX** — 計畫第 1 點的核心前提不成立。砍掉 pm.md 中的 status line 指令後，沒有任何自動化機制可以替代。status line 會永遠停在上一個狀態。

---

**F-002**
- **位置**：計畫第 1 點 + pm.md:19-34 + pm.md:40-49 + pm.md:187-193 + pm.md:349-355 + pm.md:473-479
- **嚴重度**：🔴
- **問題**：計畫宣稱 status line 指令「分散且重複」，但核查發現各處的指令內容不同，不是重複而是必要的分散
- **驗證**：逐一比對各處 status line 呼叫：
  - 第 19-34 行（表格）：列出 8 種組合（pm/sync/review/bye × running/done），是**索引表**
  - 第 43 行（`/pm` Step 0）：`reset && pm running`，是唯一有 `reset` 的地方
  - 第 48 行（`/pm` Step 0 結束）：`pm done`
  - 第 189-193 行（`/pm sync`）：`sync running` / `sync done`
  - 第 351-355 行（`/pm review`）：`review running` / `review done`
  - 第 475-479 行（`/pm bye`）：`bye running` / `bye done`

  每個子命令呼叫的 key 不同（pm/sync/review/bye），不是「重複」而是「每個命令各自負責更新自己的狀態」。表格是索引參考。如果移除各子命令裡的具體呼叫，agent 就不知道在該步驟要呼叫什麼。
- **裁決**：**MUST FIX** — 「重複」的宣稱錯誤。這些是不同 key 的不同呼叫，不能合併。

### 🟠 中等 (SHOULD FIX)

---

**F-003**
- **位置**：計畫第 4 點（共用邏輯重複 — 只寫一次，用引用）
- **嚴重度**：🟠
- **問題**：「引用」在 Claude Code skill 檔案中的行為未定義，可能不生效
- **驗證**：pm.md 是一個 slash command skill 檔案。Claude Code 讀取 skill 時，會把整個 .md 檔案內容作為 prompt 的一部分注入。Markdown 的 `[引用][ref]` 或 `[見上方](#anchor)` 語法在純文字 prompt 注入場景下沒有超連結行為 — agent 看到的是純文字。如果計畫的「引用」是指 markdown anchor link，agent 不會跳轉，只會看到一個死連結字串。如果是指「在文件內寫『見共用邏輯參考』」這種文字指引，agent 需要在同一檔案內找到對應段落，這可行但增加了出錯概率（agent 可能忽略指引直接執行）。
- **裁決**：**SHOULD FIX** — 需要明確定義「引用」的實現方式。如果只是純文字指引，要評估 agent 遵從率。

---

**F-004**
- **位置**：計畫第 2 點（Dashboard 格式寫了兩遍 — 合併成一段規則）
- **嚴重度**：🟠
- **問題**：Dashboard 格式寫兩遍是刻意的設計，不是疏忽重複
- **驗證**：pm.md:305-326 有兩個 code block，一個是 replace 格式（不含 `##` header），一個是 append 格式（含 `##` header）。兩者的差異是 pm.md 最關鍵的防 bug 規則之一（見 329 行警告和 633 行規則）。歷史 commit `3762d6b` 的 message 就是 "fix: Dashboard replace must exclude header to prevent duplicates"，說明這個區分曾經造成過 bug。合併成一段規則後，如果表述不夠明確，agent 很可能再次犯同樣的錯（在 replace 時帶上 header 導致重複）。分開寫兩個模板是最不容易出錯的做法。
- **裁決**：**SHOULD FIX** — 合併有回歸風險。如果要合併，必須驗證 agent 在合併後的表述下不會犯 replace+header 的錯誤。

---

**F-005**
- **位置**：計畫第 3 點（輸出模板佔太多空間 — 改成 bullet list）
- **嚴重度**：🟠
- **問題**：完整模板的存在是為了讓 agent 精確複製格式，改成 bullet list 會降低輸出一致性
- **驗證**：pm.md 中的 code block 模板（如 54-58 行的問候模板、149-164 行的 status 模板、551-569 行的收工摘要模板）是給 agent 的格式範本。Agent 看到 code block 會傾向直接套用，看到 bullet list 描述則需要自行組裝格式，這會導致每次執行的輸出格式不一致。對於與 Slack Canvas 互動的格式（如 306-326 行），格式精確性更加關鍵 — 多一個空行或少一個 emoji 都可能影響 Canvas 渲染。
- **裁決**：**SHOULD FIX** — 用戶面向的問候模板可以簡化，但 Slack Canvas/Dashboard 的格式模板不應該改成 bullet list。需要區分哪些模板可以簡化、哪些不行。

---

**F-006**
- **位置**：計畫整體（預估從 647 行壓到 ~400 行）
- **嚴重度**：🟠
- **問題**：行數估算過於樂觀，實際可壓縮空間遠小於宣稱
- **驗證**：逐點估算：
  - 第 1 點（砍 status line）：索引表 19-34 = 16 行，各子命令的 Step 0 段落共約 30 行。但如 F-001 所述 hook 方案不可行，這些不能砍。淨節省 = 0 行。
  - 第 2 點（合併 Dashboard 格式）：305-326 = 22 行可合併為約 15 行，但如 F-004 所述有回歸風險。最多省 7 行。
  - 第 3 點（模板改 bullet）：假設只改安全的模板（問候、status、摘要），約省 20-30 行。
  - 第 4 點（共用邏輯引用）：583-638 的共用邏輯參考有 56 行，但這段本身就是「寫一次」的設計，各子命令並沒有重複這段內容，只是引用它。真正可消除的重複很少。
  - 第 5 點（status/resume 簡化）：142-182 = 41 行，壓到 10 行省 31 行。

  樂觀估算最多省 60-70 行（假設第 1 點找到替代方案），從 648 行到約 580 行。要到 400 行需要砍掉 248 行，這意味著需要刪除約 38% 的內容，遠超上述可壓縮量。
- **裁決**：**SHOULD FIX** — 400 行的目標不現實，應修正為合理預期（550-580 行），或識別出計畫遺漏的其他可壓縮區域。

**F-007**
- **位置**：計畫第 5 點 + pm.md:170-182（`/pm resume`）
- **嚴重度**：🟠
- **問題**：`/pm resume` 中的 `claude --resume <sessionId>` 在 Claude Code session 內執行會失敗或產生未定義行為
- **驗證**：pm.md:175-176 說「執行 `claude --resume <sessionId>`，注意：這會結束當前 session」。但在 Claude Code 的 skill 執行環境中，bash 呼叫 `claude --resume` 會嘗試啟動一個新的 Claude CLI instance，不會「結束當前 session」— 它會在子進程中啟動新 session，而當前 session 繼續運行。這是 pm.md 本身的 bug，計畫說「簡化到 3-5 行」但沒提到要修這個問題。
- **裁決**：**SHOULD FIX** — 計畫遺漏了 pm.md 中一個已知的可行性問題。

## 未覆蓋風險

計畫完全沒有提到以下 pm.md 中的問題：

1. **`/pm resume` 的 `claude --resume` 可行性問題**（見 F-007）— 這是現有 pm.md 的功能性 bug，比格式精簡更重要。

2. **`/pm sync 2` 選單編號矛盾**（見 F-009）— 快捷參數的行為描述與選單編號不對齊。

3. **review 的 status line 是死代碼**（見 F-008）— pm-update.sh 和 statusline.js 都不支援 review key。

4. **`/pm bye` Step 4 的 Session ID 撈取方法脆弱** — pm.md:521-525 假設可以從 `~/.claude/sessions/` 目錄的 JSON 檔案中透過 cwd 匹配找到 session ID，但這依賴 Claude Code 的內部實現細節，沒有任何官方 API 保證這個路徑和格式穩定。

5. **sync 選 1 的 smart-commit 是靜默執行的**（pm.md:230-231）— 自動 `git add` + `git commit` 不問使用者，但 `/pm bye` Step 3（pm.md:509）有同樣功能卻會問。同一行為在兩個地方的互動模式不一致。
