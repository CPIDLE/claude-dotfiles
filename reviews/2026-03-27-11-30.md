**F-007**
- **位置**：commands/pm.md:267-268（Step B — Apps Script URL 構建）
- **嚴重度**：🟡
- **問題**：Apps Script Web App 的 URL 構建方式是把 base64 payload 放在 query string 中（`?action=replace_section&payload=<BASE64>`）。GET 請求的 URL 長度有限制（大多數瀏覽器和 server 限制 2000-8192 字元）。當 progress.md 內容較長時，base64 編碼後的 payload 很容易超過此限制，導致 Apps Script 返回錯誤或截斷。pm.md 未定義此邊界條件的處理方式。
- **驗證**：一個 5KB 的 progress.md 經 base64 編碼後約 6.8KB，加上 URL 前綴和 action 參數，總 URL 長度約 7KB。Google Apps Script 的 GET 請求 URL 限制為 2KB（doGet），超過會被截斷。pm.md:269 的錯誤處理只檢查 `"status":"ok"`，不處理因 URL 過長導致的靜默截斷。
- **裁決**：NICE TO FIX — 應考慮改用 POST 請求或記錄 payload 大小限制。

## 未覆蓋風險

1. **Apps Script doGet URL 長度限制**：pm.md 使用 GET + query string 傳遞 base64 payload，Google Apps Script 的 doGet URL 限制遠小於大型 progress.md 的 base64 編碼結果。整個 Google Doc 同步功能在大型專案中可能靜默失敗。

2. **`gmail-work` MCP 的 `gmail_list_emails` 防洗版查詢可靠性**：防洗版機制依賴查詢 sent mail（`to:cpidle-project-report@gyro.com.tw`），但 Gmail 搜尋在 sent folder 的索引可能有延遲（幾秒到幾分鐘）。若 sync 後立即再次 sync，防洗版可能因索引延遲而失效，導致重複發送。

3. **`/pm bye` 的 Step 5 引用「選 1 的完整流程」但 Step PRE 不在 bye 路徑中**：如果 agent 從 `/pm bye` 進入而非 `/pm sync`，Step PRE 的 Google 工具可用性檢查定義在 `/pm sync` 區段下。agent 是否會回頭讀取 `/pm sync` 的 Step PRE 取決於 agent 的理解能力，文件未明確串接。

4. **bye.md 與 pm.md bye 步驟編號差異**：bye.md 有 Step 0-9（10 步），pm.md bye 有 Step 0-8（9 步）。bye.md 缺少 pm.md 的 Step 2（easy 審核）和 Step 6（Retro to Memory），但多了 Step 5（檢查 Git Remote）。兩個 bye 路徑的功能集不同，使用者若以為等價會遇到差異。

5. **Apps Script Web App ID 暴露在公開 repo 中**：CLAUDE.md 包含完整的 Apps Script deployment ID（`AKfycbz...`），此 repo 為 public（GitHub: CPIDLE/claude-dotfiles）。雖然 Apps Script Web App 通常有自己的權限控制，但 deployment ID 暴露在公開 repo 意味著任何人都能嘗試呼叫此 endpoint。
