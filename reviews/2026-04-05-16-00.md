## Findings

### MUST FIX

---

**F-001**
- Location: `do_helper.py:148-154` (`parse_review`)
- Severity: MUST FIX
- Issue: JSON parse failure silently defaults to `{"verdict": "pass"}`, masking broken reviews
- Verification: If the review model returns malformed JSON (which is common with LLMs), `parse_review` catches `JSONDecodeError` and returns a verdict of `"pass"`. This means a garbled or incomplete review response is treated as "code passed review". In `cmd_deep` line 196, `verdict == "pass"` causes the flow to skip Round 3 entirely and emit the unreviewed R1 code as the final result. The whole point of deep mode's multi-round flow is to catch issues -- silently passing on parse failure defeats this.
- Verdict: MUST FIX. At minimum, the fallback verdict should be `"fail"` with an issue noting the parse failure, so Round 3 is triggered. Or emit a warning in the JSON-lines output so Claude Code can report it.

---

**F-002**
- Location: `do_helper.py:181-182` and `do.md:67-71`
- Severity: MUST FIX
- Issue: `do_helper.py` emits `{"round":1,"status":"started"}` lines that `do.md` does not document in its output format spec
- Verification: `cmd_deep` emits `{"round":1,"status":"started"}` at line 181, `{"round":2,"status":"started"}` at line 187, and `{"round":3,"status":"started"}` at line 203. But `do.md` lines 67-71 only document `"status":"done"` lines. Claude Code is instructed to "parse the last line" for the final result, so `started` lines may not cause breakage, but the documented format is inconsistent with actual output. If Claude Code or any downstream consumer does strict validation, unexpected JSON keys/values will cause errors.
- Verdict: MUST FIX. Either suppress the `started` emissions, or document them in `do.md`.

### SHOULD FIX

---

**F-003**
- Location: `do_helper.py:70`
- Severity: SHOULD FIX
- Issue: `GeminiEngine.__init__` uses `os.environ["GEMINI_API_KEY"]` (KeyError on missing), but `cmd_check` catches this as a generic Exception
- Verification: `cmd_check` at line 161-168 instantiates the engine inside a try/except and converts the error to `{"ok":false,"error":"..."}`. This works. However, when running `easy` or `deep` commands, the engine is also instantiated at line 254 with a try/except. The `KeyError` message for a missing env var is just `'GEMINI_API_KEY'` -- a bare string with no context. The error message emitted would be `"Engine init failed: 'GEMINI_API_KEY'"` which is cryptic.
- Verdict: SHOULD FIX. Add a descriptive check: `os.environ.get("GEMINI_API_KEY") or raise ValueError("GEMINI_API_KEY not set")` or similar.

---

**F-004**
- Location: `do_helper.py:83-84`
- Severity: SHOULD FIX
- Issue: Ollama prompt injection via `prompt` field -- user-controlled spec is concatenated into the prompt with no sanitization
- Verification: All three engines concatenate the spec directly into the prompt string. For Gemini and Ollama, this goes directly to an API. For `OpencodeEngine`, the spec is passed via stdin. While prompt injection in LLM-to-LLM delegation is a known accepted risk in this architecture, the Ollama engine sends the prompt via JSON body to a local REST API. If `OLLAMA_URL` is pointed at a non-Ollama endpoint, the raw spec (which could contain arbitrary content from a user task description) is sent to that endpoint. No URL validation is performed on `OLLAMA_URL`.
- Verdict: SHOULD FIX. At minimum, validate that `OLLAMA_URL` is a localhost/private address, or document the trust assumption.

---

**F-005**
- Location: `do_helper.py:89`
- Severity: SHOULD FIX
- Issue: Ollama `urlopen` timeout is 180s, but `do.md` sets Bash timeout at 120000ms (120s) for easy mode
- Verification: `do.md` line 203 says easy mode uses `timeout: 120000` (2 minutes). `OllamaEngine.call` at line 89 uses `timeout=180` (3 minutes). If Ollama is the engine for an easy call, the Bash timeout will kill the process before the urllib timeout fires, resulting in a non-JSON error (killed process) instead of a clean timeout message. The same issue exists for `OpencodeEngine` at line 101 (`timeout=180`).
- Verdict: SHOULD FIX. Either align the Python-side timeouts to be shorter than the Bash-side timeout, or handle SIGTERM/process kill gracefully.

---

**F-006**
- Location: `do_helper.py:98-112`
- Severity: SHOULD FIX
- Issue: `OpencodeEngine.call` ignores stderr and non-zero exit codes
- Verification: `subprocess.run` captures both stdout and stderr, but the code only processes `stdout`. If `opencode run` fails (non-zero return code), `result.returncode` is never checked. The method would return an empty string (no JSONL parts matched), which then gets used as if it were a valid LLM response. In easy mode, this empty string becomes `{"status":"ok","result":""}`. In deep mode, it silently feeds an empty string into the review round.
- Verdict: SHOULD FIX. Check `result.returncode` and raise on failure.

## Uncovered Risks

1. **No integration test**: There is no test file for `do_helper.py`. The three engine classes, `parse_review`, and the multi-round `cmd_deep` flow have zero test coverage. Given that `parse_review` has a silent-pass bug (F-001), this is concerning.

2. **No rate limiting / cost control**: `cmd_deep` makes 2-3 API calls per invocation. There is no guard against rapid repeated invocations accumulating API costs. The `do_level.md` mechanism controls task complexity but not invocation frequency.

3. **Large spec + large response = memory**: The deep flow concatenates spec + R1 output + R2 output + issues + fix instructions into R3's prompt. If R1 produces very large output (e.g., a large module), the R3 prompt could exceed model context limits. No size check or truncation exists.

4. **Windows path handling**: `do.md` uses `$HOME/.claude/skills/do/scripts/do_helper.py` with Unix path separators. On Windows (the current platform), `$HOME` in Git Bash expands correctly, but if Claude Code invokes this outside of Git Bash (e.g., via cmd.exe or PowerShell), the path would fail. The install script uses PowerShell `Copy-Item` which handles this, but runtime invocation assumes bash.

## Summary

The refactoring from inline `python -c` to a dedicated `do_helper.py` is architecturally sound and the code is clean. However, two blocking issues exist: the silent-pass on review parse failure (F-001) undermines deep mode's core value proposition, and the undocumented `started` emissions (F-002) create a contract mismatch between do.md and do_helper.py. Four additional issues around error handling (F-003, F-005, F-006) and input validation (F-004) warrant attention before merge.
