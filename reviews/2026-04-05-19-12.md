## Findings — Code Review

### F-001: No consumer exists for the quota-aware rule
- **Location**: `global-claude.md:59-64`
- **Severity**: MUST FIX
- **Problem**: The rule says "read `~/.claude/usage-cache.json` and check `session.utilization >= 80`", but this is a natural-language instruction to Claude Code (an LLM). There is no code that enforces it. Claude Code must interpret this text, read a file, parse JSON, and compare a number — every single session, on every deep-eligible task. If Claude Code misinterprets, skips, or hallucinates the check, the rule silently fails. There is no fallback, no log, no verification that the check actually happened.
- **Verification**: Searched entire repo for any script/hook that reads `usage-cache.json` for delegation decisions. Only `statusline.js` writes it; nothing else reads it programmatically. The rule is purely prompt-based.
- **Ruling**: MUST FIX — a quota-aware rule that depends on LLM compliance with zero enforcement is not quota-aware; it is quota-hopeful.

### F-002: Cache staleness window allows stale decisions
- **Location**: `statusline.js:9` (`REFRESH_SEC = 300`) vs `global-claude.md:64`
- **Severity**: MUST FIX
- **Problem**: `usage-cache.json` is refreshed every 5 minutes by the statusline, but only when the statusline runs. If the user hasn't triggered a statusline refresh recently, the cache could be arbitrarily old. The rule says "cache not found or read failure = green zone (no delegation)" but says nothing about stale cache. A cache from 2 hours ago showing 40% utilization would be treated as green even if current utilization is 95%.
- **Verification**: `statusline.js:14` — cache validity is `Date.now() - c.ts < REFRESH_SEC * 1000` (5 min), but this is only checked by statusline itself for its own refresh logic. The delegation rule in `global-claude.md` has no staleness check.
- **Ruling**: MUST FIX — stale cache defeats the purpose of quota-awareness.

### F-003: Threshold value 80 is arbitrary and undocumented
- **Location**: `global-claude.md:62`
- **Severity**: SHOULD FIX
- **Problem**: `>= 80` is the threshold for "red zone" delegation. No rationale is given for why 80 and not 70 or 90. The statusline uses the same threshold for red coloring (`statusline.js:53`: `if (pct >= 80) return '\x1b[31m'`), so there is visual consistency, but the delegation decision has much higher stakes than text color. At 79% utilization, the system self-handles a deep task that will push utilization higher; at 80%, it delegates to save quota. The boundary behavior is not discussed.
- **Verification**: Cross-referenced `statusline.js:53` color thresholds with `global-claude.md:62` delegation threshold. Same number, different consequences.
- **Ruling**: SHOULD FIX — document the rationale or make the threshold configurable.

### F-004: "Faster 4x" claim is unsubstantiated in commit
- **Location**: `global-claude.md:63` — "Claude Code 自己處理（更快 4x、品質一次到位）"
- **Severity**: SHOULD FIX
- **Problem**: The commit message says "benchmark verified" but the benchmark data (3s easy, 54s deep for /do vs 10s, 13s for Claude Code) yields speed ratios of 0.3x (easy, /do is faster) and 4.15x (deep, Claude is faster). The "4x" claim cherry-picks the deep result and ignores that for easy tasks, /do is actually faster. The phrase "更快 4x" without qualification is misleading.
- **Verification**: Session summary states /do easy = 3s, /do deep = 54s, Claude easy = 10s, Claude deep = 13s. 54/13 = 4.15x for deep only.
- **Ruling**: SHOULD FIX — either qualify "deep tasks: ~4x faster" or remove the speed claim.

### F-005: Easy tasks "always delegate" contradicts speed data
- **Location**: `global-claude.md:66-70`
- **Severity**: SHOULD FIX
- **Problem**: Easy tasks are marked "always delegate" to /do, yet the session benchmark shows Claude Code (10s) is slower than /do easy (3s) by only 7 seconds — and the quality difference was not measured. The rationale for always delegating easy is presumably "save quota", but this is never stated. If it is about quota savings, say so. If it is about speed, the data only marginally supports it.
- **Verification**: Session data: /do easy 3s vs Claude 10s.
- **Ruling**: SHOULD FIX — state the rationale explicitly (quota savings, not speed).

### Context Waste Findings

#### CW-001: Self-benchmarking method is flawed and expensive
- **Severity**: SHOULD FIX (process)
- **Problem**: Using Claude Code to write email-validator and rate-limiter code, then measuring wall-clock time, conflates "time to generate code" with "time for Claude Code to process the conversation turn". The 10s and 13s include tool call overhead, file writes, and syntax checks — not pure generation time. Meanwhile /do's 3s and 54s include Python startup + API latency. The comparison is apples-to-oranges. Worse, Claude Code spent ~20% of session context generating throwaway benchmark code.
- **Better approach**: Simply time the wall-clock from prompt to completion for equivalent /do and Claude Code tasks in separate sessions. Or use existing benchmark.py (which already exists in the repo!) with properly designed tasks. Zero code generation needed.

#### CW-002: Plan mode for a one-paragraph edit
- **Severity**: SHOULD FIX (process)
- **Problem**: The session entered full plan mode (EnterPlanMode -> design -> ExitPlanMode -> approval -> execute) for a task that ultimately changed one section (~15 lines) of one file. The plan-execute workflow specifies "multi-file / architectural / multi-step tasks". This was a single-file, single-section text edit. The plan mode ceremony consumed ~15% of context for a task that could have been proposed inline: "I'll replace the /do delegation section with quota-aware rules. Here's the draft: [draft]. Approve?"
- **Threshold question**: What is the minimum task size that justifies plan mode? This session suggests the bar is too low.

#### CW-003: Edited wrong file first
- **Severity**: SHOULD FIX (process)
- **Problem**: The session edited `~/.claude/CLAUDE.md` directly, then discovered that `global-claude.md` in the repo is the source of truth (install script copies it). This required re-doing the edit on the correct file. The install script relationship (`global-claude.md` -> `~/.claude/CLAUDE.md`) is documented in the repo's own CLAUDE.md and install.ps1. A single Grep for "global-claude" or reading the install script first would have prevented this.
- **Token cost**: ~10% of session wasted on editing the wrong file + discovering the error + re-editing.

#### CW-004: Install script ran twice
- **Severity**: NICE TO FIX (process)
- **Problem**: `install.ps1` was run once after editing `~/.claude/CLAUDE.md` (unnecessary — that file IS the install target) and once after editing `global-claude.md` (correct). The first run was entirely wasted.
- **Token cost**: ~5% of session on the unnecessary install + its output.

### Ideal Flow (If Starting Over)

1. **Skip self-benchmarking** — use wall-clock timing or existing `benchmark.py`. Save ~20% context.
2. **Skip plan mode** — propose the edit inline. Save ~15% context.
3. **Read install.ps1 first** — understand source-of-truth is `global-claude.md`. Save ~10% context.
4. **Edit `global-claude.md` directly** — single edit, single install, single commit. Save ~5% context.
5. **Total estimated savings**: ~50% of session context.

Ideal token budget for this task: ~50% of what was actually spent. The task was fundamentally "add 15 lines to a config file based on 4 data points from a benchmark."

## Uncovered Risks

1. **No integration test**: The quota-aware rule has never been tested end-to-end. There is no way to verify that Claude Code actually reads the cache, parses it correctly, and makes the right delegation decision. The first real test will be in production.
2. **API schema drift**: `usage-cache.json` structure depends on the undocumented Anthropic OAuth usage API (`/api/oauth/usage`). If the API response changes field names, the cache will have different keys than what the delegation rule expects.
3. **Race condition**: If statusline refreshes cache while Claude Code is mid-read, the JSON could be partially written. `statusline.js` uses synchronous `writeFileSync` which mitigates this on the write side, but the reader (Claude Code, interpreting natural language) has no atomic read guarantee.
