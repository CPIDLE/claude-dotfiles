# Polish — 繁中寫作精修 Skill（統一版）

> **預設模式 = 提醒**。cc 觸發後**不主動跑流程、不改檔**，只給使用者：
> 1. 量體統計（字數、章節數，可信數據）
> 2. polish 涵蓋的處理範圍清單（字句 / 段落 / 結構）
> 3. references/ 內參考規則的入口
>
> 使用者人工判讀文件，自己決定要怎麼處理：
> - 自己動手改 → cc 不介入（提醒完就退）
> - 請 cc 改某幾段 / 章 → cc 進「執行模式」走對應流程
> - 完全不改 → 退出
>
> **不要 cc 自動診斷、自動分流、自動跑 Level 1-4 或 Stage 1-3**。中文沒分詞工具，cc 跑 grep 計密度誤判率高；「字句問題嚴重 vs 結構問題嚴重」這種判斷使用者人工判讀比 cc 算的準。

## 觸發後第一步：診斷 + 分流

### 1. 檔型守門

| 條件 | 動作 |
|---|---|
| 副檔名是 code（`.py .ts .js .tsx .go .rs .java .c .cpp .sh .ps1` 等）/ commit msg / log / 純 config（`.json .yaml .toml .ini`） | **退出**：「polish 是繁中散文精修 skill，這份檔型不適用」 |
| 非繁中為主的文件 | **退出**：「polish 只處理繁中文件」 |
| `.md .txt .docx` + 內文以中文為主 | **進場** |
| `.md` 但內容是技術 release notes / API doc / changelog | **問**：「這份比較像技術文件，要怎麼處理？」 |

### 2. Resume 偵測

進場後，先看工作目錄是否有過去進度：

```bash
ls <檔案所在資料夾>/<檔名>_skeleton*.md <檔名>_partial.md <檔名>_v[0-9]*.md 2>/dev/null
```

| 找到 | 動作 |
|---|---|
| 只 `<檔>_skeleton*.md` | 問：「發現之前的 skeleton 進度（v?），要 resume Stage 2 還是從頭？」 |
| 只 `<檔>_partial.md` | 看 partial 內容是否含「§N 完成」進度紀錄 → 問：「上次跑到 §N，要從 §N+1 接續還是從頭？」 |
| `<檔>_skeleton*.md` + `<檔>_partial.md` 同時存在 | 問：「同時發現 skeleton 和 partial，上次跑到流程 B 哪一階？（Stage 2 中？Stage 3 整合中？）」並列出兩檔最後修改時間 |
| `<檔>_v[N].md` | 列出告知，不主動 resume（顯式版本由使用者管理）|
| 無 | 從頭開始 |

### 3. 量體（純機械統計，可信）

跑使用者可信的 deterministic 計算：

| 項目 | 怎麼算 |
|---|---|
| 字數 | 中文字 + 英文 + 數字 + 標點，`wc -m` 之類 |
| 章節數 | grep `^## ` 行數 |
| 是否含開頭摘要 | 第一個 H2 標題是否為「摘要 / TL;DR / Summary / Overview」 |

**不跑** grep 計贅字 / AI 兵符 / 文謅謅密度——中文沒分詞工具，「進行」會命中「進行式 / 進行中」，誤判率高。**字句層問題人工判讀比 cc grep 準**。

### 4. 提醒使用者：polish 處理範圍清單

cc **不下印象、不推薦流程**，列出 polish 涵蓋的處理項，請使用者自己判斷文件有哪些問題：

```
[Polish 觸發] <檔名>
量體：X 字 / Y 個 H2 章節 / 開頭摘要：[有 / 無]

cc 不會自動跑 grep 計贅字密度（中文沒分詞工具，誤判率高）。
請使用者人工判讀文件，從下面清單挑要處理的：

字句層（A）
□ AI 機器味 / 清喉嚨詞（「值得注意的是」「綜上所述」「不可否認」）
□ 中文贅字（「進行 X」「相關的」「的部分」「方面」「其實」「事實上」）
□ 官腔長句（長串名詞化、抽象主詞「相關研究指出」）
□ 文謅謅（「揭櫫」「蘊含」「鑒於」「抑或」）
□ 英文縮寫章節標題（TL;DR / FAQ / FYI 改中文）
□ 動詞無力（「進行討論」「給予建議」改具體動作）

段落層（A 內含）
□ topic sentence 缺失（段落第一句沒交代本段主旨）
□ 段落間承接不順（並列三段沒邏輯關係）
□ 過度說教式收尾（「綜上所述，這顯示了……」）
□ 三段同樣開頭（「首先 / 其次 / 此外」是 AI 兵符）

結構層（B）
□ 核心概念散落多章（同一概念定義在多處）
□ TL;DR / 摘要 跟內文不對齊（摘要說的內文沒展開）
□ 章節順序混亂或重複
□ 論點跨章重複
□ 章節該合該砍（過多細節該移附錄）

可選流程
- 流程 A（字句層 4 levels：文件 → 段落 → 句子 → 字詞）
- 流程 B（結構重組三階段：skeleton → 填空 → 整合）
- 流程 C（B 先 A 後串聯）
- 自訂：使用者直接指出哪幾段 / 哪幾章要改

請告訴 cc：
1. 哪幾個是這份的主要問題？（從清單勾或自描述）
2. 想優先走哪個流程？或先處理哪幾段？
```

使用者選了 → cc 走對應流程。**cc 不要替使用者決定哪些問題嚴重**。

### 5. 短文 / 已 OK 文件處理

| 量體 | 動作 |
|---|---|
| < 1000 字（短文，~50-100 行）| 不跑完整流程，直接讀整檔 → 列建議改寫位置給使用者選 |
| 使用者看完清單說「沒什麼好改的」 | 告知「文件看起來 OK，建議列具體想改的段落，避免無病呻吟改壞」並停止 |
| 1000-3000 字（中文）| 看使用者選的流程跑，但流程 A/B 可簡化（例如 A 合併 Level，B Stage 1 skeleton 縮小）|
| > 3000 字（長文）| 完整流程 |

### 6. 副本規則（不論流程都適用）

說明：
- 「使用者明確要 v0/v1/v2」= 使用者**主動**講「我要 v0 / v2 副本」「保留原檔我看 v 版」這類話。cc 推薦或 cc 自己決定不算「明確」
- 若 cc 想保留原檔但使用者沒明說 → 應該**先問**「要不要保留原檔開副本？」拿到同意才寫副本

| 情境 | 副本？ | 命名 |
|---|---|---|
| **預設精修** | ❌ 不建副本 | 直接改原檔，git history 是版本記錄 |
| **使用者明確要 v0/v1/v2** | ✅ 顯式接受 | 寫到 `<原檔>_v0.md`，使用者意圖優先 |
| **流程 A 分段模式 / 流程 B 中間進度** | ✅ 過程檔 | `<原檔>_partial.md` 或 `<原檔>_skeleton.md`（resume point，非版本副本） |
| **AI 自作主張的 `_final` `_working` `_v2`** | ❌ 嚴禁 | 不要為了「保險」自建未經授權的副本 |

判準：「使用者沒說要副本」 + 「不是流程過程檔」 → 不建副本。

### 7. Git 保護

- 檔在 git repo 外 → 提議 `git init`，等使用者同意
- 在 repo 內 → 直接 `git add <file> && git commit -m "polish: 原稿快照"`
- 各階段完成後 commit：`polish: <流程> <階段> 完成`

## 進入執行模式的前提

以上是**提醒模式**（預設）：cc 列清單給使用者看，使用者自己改。

**只有使用者明確說「請 cc 幫我改 X」「請 cc 跑流程 A」這類**才進**執行模式**，走下面的流程 A / B / C。

執行模式啟動條件（任一）：
- 使用者明確指定要改的範圍（「改 §3 段的贅字」「重組 §四章節」）
- 使用者明確選流程（「跑流程 B」「走 skeleton 流程」）
- 使用者明確要 cc 動手（「幫我改」「請動」「直接改」）

**模糊指令拒絕當作執行模式**（如「整理一下」「弄好」「處理」）— cc 反問「要 cc 動手改哪幾段，還是只給建議讓你自己改？」

## 流程 A：字句層（Level 1-4）— 執行模式

**適用**：使用者明確要 cc 改字句，且結構 OK。**強制四層 top-down**，每層完成等使用者核准再進下一層。

### A 核心方法：刪除測試

每層共通原則（源自 Becker）：**砍掉這段，意思變了嗎？沒變就刪。**

不要先想「這段為什麼要留」，要想「這段沒了會怎樣」。

### A.1 Level 1：文件層

跳過句子先看大架構：

1. **開頭抓不抓得住人** — 第一句是公式化開場（「在現代社會」「隨著……的發展」）還是直接切入？
2. **標題** — 短而精準，還是塞滿冒號和子標題？**繁中文件章節標題不用英文網路縮寫**：

   | 砍 | 改 |
   |---|---|
   | TL;DR | **摘要**（預設）/ 核心 / 重點 / 一頁讀完 / 概要 |
   | FAQ | 常見問題 |
   | FYI | 供參 / 給你參考 |
   | ASAP | 盡快 / 最遲 X 時前 |
   | BTW | 順帶一提 / 補充 |
   | TBD | 待定 |
   | TBA | 待公布 |
   | AKA | 即 / 也就是 |
   | ICYMI | 補充說明 |

   章節標題（H1-H4）一律改中文；內文若是技術用語（API、SDK、CI/CD、SRE）保留。

3. **章節結構** — 標題承諾的內容跟實際段落對不對得上？
4. **論點是否重複** — 同一個 point 出現在兩個地方？合併或刪一個。**注意**：若診斷顯示重複跨 ≥ 4 章 → 這是結構問題，跳到流程 B
5. **是否有具體例子** — 每個抽象概念至少一個具體 case
6. **段落是否並列無連接** — 三個段落像條列堆疊、沒承接邏輯

→ 提議改動，等使用者核准再進 Level 2。

### A.2 Level 2：段落層

1. **每段一個主旨**
2. **topic sentence** — 段落第一句說明本段在幹嘛
3. **段落結尾不要說教** — 「綜上所述，這顯示了……」八成冗餘
4. **段落間要有承接** — 並列三段時 B 段開頭該交代為什麼接在 A 後面
5. **三段同樣開頭 = 改寫** — 「首先 / 其次 / 此外」是 AI 兵符
6. **三點清單檢查** — 第三點是真有意義還是湊數

→ 提議改動，等使用者核准再進 Level 3。

### A.3 Level 3：句子層

1. **長串名詞化** — 「進行 X 的處理」→「處理 X」
2. **抽象主詞 → 具體主詞** — 「相關研究指出」誰指出？
3. **「這」「該」單獨指代** — 改成「這個現象」「這份資料」
4. **句子主謂距離 > 15 字 = 重寫**
5. **動詞無力 = 換掉** —「進行 / 實施 / 給予」換具體動作
6. **死掉的比喻** —「百花齊放」「再上一層樓」
7. **後設評論** — 「以下我將說明」直接做不要說
8. **過度自評** —「這是非常重要的觀點」自誇無效
9. **破折號濫用** — 改兩個短句、冒號、或拉進主句

→ 提議改動，等使用者核准再進 Level 4。

### A.4 Level 4：字詞層

**進入 Level 4 時 Read `references/level4-word-rules.md`**，內含中文贅字完整對照表（11 類規則）。

最後拋光，砍清喉嚨詞、路標詞、評價副詞、空洞強調、能力詞、文謅謅、過度讚美、英文縮寫。

→ 提議改動，等使用者核准定稿。

### A 分段模式（≥ 300 行 / ≥ 8K 字）

1. TaskCreate 每章節一個 task
2. 全文先做 Level 1 一次（章節結構 / 重複論點 / 順序）
3. 接著按 section 跑 Level 2-4（逐段密集處理）
4. 每段寫到 `<原檔>_partial.md`（累積，不每段一檔）
5. 全文完成做 Level 1 收尾（補 missed 跨段問題）
6. 進度回報格式：「§N 完成，砍 X 行（從 Y → Z 行），累計 N/M 段」
7. 完成後 rename `_partial.md` → `_v[N].md` 或就地覆寫原檔（依使用者意圖）

## 流程 B：結構層（Stage 1-3 skeleton）— 執行模式

**適用**：使用者明確要 cc 重組大綱（核心概念散落、章節該重組、TL;DR 跟內文不對齊）。**Stage 1 必過使用者核准才進 Stage 2**。

### B.1 Stage 1：Skeleton（先談架構，不寫內文）

**產出檔**：`<原檔>_skeleton.md`（2-3 頁），含五件事：

| 元件 | 內容 |
|---|---|
| **TL;DR** | 核心主張 1 句 + 架構圖 1 張 + 5-7 條 bullet |
| **章節 outline** | H2 / H3 標題 + 每節「要回答什麼」+「不回答什麼」對照表 |
| **概念地圖** | 5-10 個核心概念，每個指定一個 home 章節，他處只 cross-ref |
| **讀者旅程** | 5 / 30 / 全讀三種角色該讀哪幾節 |
| **結構性砍除清單** | 對應原文哪些段落 / 章節是冗餘 / 跨章重複 / 該重置 |

詳細範本見 `references/skeleton-template.md`。

#### Stage 1 強制 checkpoint

skeleton 寫完**必須**等使用者核准才進 Stage 2：

1. 寫到 `<原檔>_skeleton.md`
2. 文末列「使用者請審查」3-5 個關鍵問題
3. 停下來等回應，不自動進 Stage 2

#### 預期迭代

skeleton 通常要 3-5 版才對齊核心主張、概念地圖、讀者旅程。**多版本是正常**——常常使用者反饋暴露的是 framework 本身要重整（例：L1-L3 stack 切法錯，要改 L0-L3 risk spectrum）。

寫到：`<原檔>_skeleton.md`、`_skeleton_v2.md`、`_v3.md`...直到使用者明確核准。

### B.2 Stage 2：Section 填空（逐章獨立完成）

skeleton 核准後：

1. TaskCreate 每章一個 task
2. 每段填空時 import 兩條約束：
   - 「這節要回答什麼」（防偏題）
   - 「這節不能講什麼」（防侵入他節 turf）
3. 每段寫到 `<原檔>_partial.md`（累積，不每段一檔）
4. 每段完成 self-check：
   - 提到核心概念，本節是 home 嗎？不是 → 砍剩 cross-ref
   - 跑出 skeleton 沒列的新概念？停下來問使用者
   - 跟前段是否重複？有 → 砍

**取材策略**：80% 沿用原文（表格、技術細節、工時數字直接搬），20% 重新組織（概念位置調整、術語統一、cross-ref 重連）。**不要為了重寫而重寫**。

進度回報：「§N 完成，原文 X 行 → Y 行（-Z%），累計 N/M 章，partial 已寫」

### B.3 Stage 3：Cohesion Check（全文整合驗證）

Read `references/cohesion-checks.md` 跑驗證清單：

- 自動 grep 驗證（核心概念 single source、術語一致、cross-ref 連結、舊術語殘留）
- 人工驗證（TL;DR ↔ 內文對齊、5 分鐘版實測、章節銜接）
- 整合報告（規模變化、skeleton 迭代次數、結構性砍除統計）

完成後 rename `<原檔>_partial.md` → `<原檔>_v[N].md`。

## 流程 C：串聯（B 先 A 後）— 執行模式

**適用**：使用者明確要 cc 結構重組 + 字句兩者都改。

### C 流程

1. 先跑流程 B（結構重組）至 Stage 3 完成
2. **B 完成後強制 checkpoint**（不要靜默跳過 A）：

   ```
   [流程 C 中段] 結構重組完成
   - 規模：原文 X 字 → Y 字（-Z%）
   - skeleton 迭代 N 版
   - 結構性砍除完成 M 處

   是否繼續流程 A 字句層？
   - 是 → 從 Level 2 起跑（Level 1 結構已在 B 處理過，跳過）
   - 否 → 此處停，使用者自己 review 結構版本
   ```

3. 走流程 A 時可**簡化合併** Level 2-4 為兩輪：
   - 第一輪：Level 2 段落 + Level 3 句子（合併處理段落內邏輯 + 句子改寫）
   - 第二輪：Level 4 字詞層
   - 兩輪而非四輪，減少 checkpoint 互動疲勞

4. 最後一次性 commit / rename

**不要混跑**（同一段內又改結構又改字句）— 注意力切割成本太高，容易兩邊都不到位。

### Checkpoint 預算

| 流程 | checkpoint 數 |
|---|---|
| 流程 A 完整 | 4 個（Level 1-4 各一）|
| 流程 A 簡化（中文 1000-3000 字）| 2 個（Level 1-2 / Level 3-4）|
| 流程 B 完整 | 3 個（Stage 1-3 各一，Stage 1 可能多版迭代）|
| 流程 C 完整 | 3 + 2 = 5 個（B 三 + 中段 1 + A 簡化兩輪 = 5；但實質上 A 兩輪可合一輪 = 4）|

5+ checkpoint 會互動疲勞，盡量壓在 4 以下。

## 校準（避免過度承諾）

**目標**：
- 字句層 → 讀起來有人味、敢下判斷、夠具體
- 結構層 → 骨架清楚、概念集中、讀者依角色取用

**不是目標**：完美。微微的不平滑反而像真人寫的；骨架過度精雕反而僵化。

**遇到不可能完成時要說**：

| 情境 | 處理 |
|---|---|
| 原文核心主張本身有邏輯洞 | 退場：「這是論述問題不是寫作問題，polish 改不了核心錯誤。建議跟使用者討論論點再回來」|
| 使用者要的是「全新文件」（不是精修現有）| 退場：「polish 是精修工具，從零起草建議用其他寫作 skill」|
| 跑 5 版 skeleton 還對不齊 | 停下來問：「我的理解可能 fundamentally 錯了，能用 100 字概括這份要講什麼？」|
| 自動診斷信號不強但使用者堅持要改 | 答：「文件診斷顯示問題不大，建議列具體想砍的段落 / 章節給我，避免無病呻吟改壞」|

**Genre 感知**：學術論文保留必要 hedging；blog 可口語；信件要短；報告要嚴謹；簡報文案一句一 point。不要把所有 genre 壓成同一個調。

## 與 cc 預設風格的關係

cc 全域 CLAUDE.md 預設「直接給結果、不複述、不加開場白」（技術回應風格）。

**`/polish` 觸發時**，技術風格讓位給寫作判斷：

- 流程 B Stage 1 階段必須跟使用者深度對話（5-10 輪迭代正常）
- 各層 / 各段完成做 self-check 後給簡短回報（不只是丟結果）
- 解釋為什麼這段該改（讓使用者學到 pattern）
- 提多個改寫選項供選（不是每次都自作主張）

但仍適用：不堆 emoji、不寫長 docstring、繁中文件章節標題不用英文縮寫。
