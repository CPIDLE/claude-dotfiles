# Stage 3 Cohesion Check 清單

**用法**：Stage 2 全部章節完成後，跑本檔的 check 清單。

---

## 自動 grep 驗證

### 1. 核心概念 single source 驗證

```bash
# 對 concept map 每個 home 概念跑 grep
grep -c "<概念 A>" <重組後檔>
# 預期：home 章節密集（在該節範圍 grep 出現多次）
# 他章只剩 cross-ref（每節 < 3 次出現）
```

判讀：
- 某非 home 章節出現概念 A 5+ 次 → 違反 single source，要砍剩 cross-ref
- 某 home 章節出現概念 A 0 次 → home 章節沒展開，要回填

### 2. 術語一致性

```bash
# 跑變體 grep，找尚未統一的別名
grep -nE "TL;DR|TLDR|tldr"     # 應該全部已改「摘要」
grep -nE "Tier I|Tier II"      # 若 framework 已換 L0-L3，這應 = 0
grep -nE "真正 A3|paradigm shift A3|無規範 A3"  # 應該只用一個術語
```

判讀：
- 出現 > 0 次的舊術語 = 殘留，要修
- 多個別名都有 = 未統一，挑一個全文取代

### 3. Cross-ref 連結正確

```bash
# 抓出所有 §X.Y 引用
grep -oE "§[一二三四五六七八九十]+|§[0-9]+\.[0-9]+" <檔> | sort -u

# 跟實際章節清單比對
grep -nE "^## |^### " <檔>
```

判讀：
- 引用的 §X 章節真的存在嗎？
- 章節編號跟引用是否一致（§3.4.5 引用但實際只有 §3.4）？

### 4. 已砍術語不殘留

跑 skeleton 第五件事「結構性砍除清單」對應的 grep：

```bash
# 例：v1 列出要砍的 12 個項目
grep -n "舊術語 1" <檔>  # 應 = 0
grep -n "舊術語 2" <檔>  # 應 = 0
```

判讀：清單上應砍的項目仍出現 = 漏砍

## 人工驗證

### 5. TL;DR ↔ 內文對齊

逐條檢查 TL;DR 5 條 bullet：

| TL;DR bullet | 內文展開位置 | 對齊？ |
|---|---|---|
| 1. [主張 1] | §X | ✓ / ✗ |
| 2. [主張 2] | §Y | ✓ / ✗ |
| ...

不對齊 = TL;DR 寫了內文沒有的 / 內文有 TL;DR 沒列的核心點 → 修哪個看判斷（通常修 TL;DR，因為內文是長期累積，TL;DR 該對齊內文）

### 6. 5 分鐘版實測

實際讀一遍只看 TL;DR + 結語，自問：

- 主張 1 句懂了嗎？
- 終點 / 範圍清楚嗎？
- 工時 / 量級拿到了嗎？
- 為何不做 X 路徑（如有的話）的理由清楚嗎？

任何一條 ✗ → TL;DR 不到位

### 7. 30 分鐘版實測

讀 TL;DR + skeleton 列出的「PM 角色該讀」幾節：

- 章節銜接順嗎？需要翻附錄才懂的內容多嗎（不該需要）？
- 概念地圖 home 章節讀過後，引用該概念的他章是否能順讀？

### 8. 章節間銜接

讀過所有章節開頭句（§X 第一段），檢查：

- 是不是「然後 / 接下來」式機械過場？
- 還是有真正的承接邏輯（從 §一 的 X 推到 §二 的 Y）？

### 9. 概念地圖實測

挑 concept map 上某 home 概念，問：

- 該概念在 home 章節定義後，他章引用時是否真的不重新定義？
- 該概念跨章使用是否一致（同一個寫法、同一個範圍）？

## 不可解 / 容忍項

不是所有問題都要在 redraft 解決：

| 情境 | 處理 |
|---|---|
| 某段技術細節太密 | 不是 redraft 該管，請使用者考慮拆附錄 |
| 某概念邊界模糊 | 在原文就模糊 → redraft 不創造清楚度，明寫「此處邊界模糊，留待後續釐清」 |
| 章節長度不均 | 不必強求均等。短章節（5-10 行）若內容該獨立就獨立 |
| 有些重複是強調 | 開頭核心主張 + 結語 recap = 2 次重複，是合理的；要砍是 polish 該管 |

**判準**：redraft 處理「結構錯」（concept distribution / 章節順序 / cross-ref），不處理「字句層」（那是 polish）也不處理「論述本身的錯誤」（那是論述者的責任）。
