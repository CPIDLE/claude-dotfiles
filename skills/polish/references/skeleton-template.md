# Skeleton 範本與設計規則

**用法**：Stage 1 寫 skeleton 時 Read 本檔。內含五件事的範本 + concept map 設計規則。

---

## 範本：skeleton 五件事

### 1. TL;DR（一頁）

```markdown
## 摘要

### 適用範圍

[一句話說清楚範圍：本文件 = ?，其他類似但不在此範圍的東西 = ?]

### 核心主張（1 句）

[最濃縮的一句話。範例格式：「X 不是 Y，要做的是 Z——理由是 W」]

### 核心架構圖

[一張 ASCII 圖。原則：垂直軸 + 水平軸 + 標出位置 + 註腳。
複雜度限制：8-12 行，超過就太複雜要再簡化]

### 關鍵主張（5-7 條 bullet）

1. **[主軸 1]** — [一句展開]
2. **[主軸 2]** — [一句展開]
（最多 7 條，超過 7 條代表還沒收攏）

### 工時 / 數字 / 量級（如適用）

| ... |
```

### 2. 章節 outline + 各節「要回答 / 不回答什麼」

```markdown
## 章節 Outline

| 章節 | 要回答的問題 | 不回答（屬其他節） |
|---|---|---|
| **§一 [標題]** | 「[問題 1]」「[問題 2]」 | 不講 [屬 §二]、不講 [屬 §四] |
| **§二 [標題]** | ... | ... |
```

**規則**：
- 「不回答」欄位是核心約束 — 寫 Stage 2 時防侵入他節 turf
- 章節數預設 3-6 個（含附錄 1-3）；超過 7 章 = 太散，應合併
- 每章「要回答」最多 3 個問題；超過 = 該節想做太多，要拆

### 3. 概念地圖（Single Source of Truth）

```markdown
## 概念地圖

| 核心概念 | Home 章節 | 他處只能 cross-ref |
|---|---|---|
| **[概念 A]** | §X | §Y / §Z 提到時用名稱不展開 |
| **[概念 B]** | §X.Y | 全文 cross-ref，不重定義 |

### 不再單獨命名（屬性化處理）

| 不再單獨 | 變成什麼 |
|---|---|
| ~~舊概念 1~~ | 變成 [新概念] 的屬性 |
```

**設計規則**（見下方「concept map 設計規則」）

### 4. 讀者旅程

```markdown
## 讀者旅程

| 角色 | 時間 | 該讀什麼 | 拿到 |
|---|---|---|---|
| **CXO / 決策者** | 5 分鐘 | TL;DR | 主張 + 量級 + 終點 |
| **PM / 技術窗口** | 30 分鐘 | TL;DR + §一 + §三 | 框架 + 工時 + 限制 |
| **工程師（執行）** | 全讀 | 全文 + 附錄 | 執行細節 + 組件清單 |
| **[特殊角色]** | 焦點讀 | §X + §Y | [特定 audience 拿到的] |
```

**驗收標準**：
- 5 分鐘版真的只看 1-2 處 → 能拿到核心
- 30 分鐘版章節順序合理（不需翻附錄）
- 每角色「拿到」欄是具體價值，不是「了解大概」這種空話

### 5. 結構性砍除清單

```markdown
## 原文 → 新版砍除清單

| 原文位置 | 內容 | 新版處理 |
|---|---|---|
| v1 §1.2 | A 階梯完整列表 | 砍剩 1 句 + 連結 §三 |
| v1 §4.4.5 | 撤回殘段 | 砍 |
| v1 §6.4 | 「與 X 一致」4 點 | 砍剩 1 句進 §一 |
```

**規則**：至少列 5 處可砍 / 重置點，否則 redraft 沒必要（可能 polish 就夠）。

## Concept map 設計規則

### Home 數上限

| Home 數 | 判斷 |
|---|---|
| 3-5 個 | 理想，骨架清楚 |
| 6-8 個 | 可接受，但要小心交互引用爆炸 |
| 9-10 個 | 警告，再思考能不能合併 |
| **>10 個** | **複雜訊號**，要強制收家 |

收家手段：
1. **概念合併**：意義相近的兩概念，合併為一個 home（例：「A 階梯定義」+「Phase 路徑」+「AI 提案週期」三個合併為一個「四階梯（含 Phase + AI 工作方式）」）
2. **屬性化**：某概念變成另一個概念的屬性註，不單獨命名（例：Tier I/II 變成 L0-L3 的安全屬性註，不單獨成概念）
3. **拆檔**：實在合不來 → 拆兩份文件（overview.md + appendix.md）

### Single source of truth 規則

- 每個核心概念**只有一個 home 章節**完整定義
- 其他章節提到時：用名稱 + 連結（`詳見 §X.Y`），不重新定義
- 例外：概念在 §一 出現作為「框架預告」OK，但具體展開仍在 home 章節
- TL;DR 提及概念 → 一句話 + 連結 home，不展開

### 屬性化處理（不再單獨命名）

某些概念在 v1 是獨立的，但分析後發現是另一個概念的屬性 — 屬性化能大幅減少 home 數。

| 模式 | 範例 |
|---|---|
| 兩個概念其實是同一條軸 | Phase 0-3 跟 A 階梯是 1:1 對應 → 合併 |
| 某概念是另一個的安全屬性 | Tier I/II 是 L0-L3 的危險性屬性 → L 屬性化 |
| 某概念是另一個的時序面向 | AI 提案週期是 A 階梯的「AI 在此階能做」欄位 → 收進階梯主表 |
| 某概念其實是某 case 的特例 | 「真正 A3」是 paradigm shift 一個 case → 收進 A 階梯尾段 |

### Cross-ref 寫法

- **同段內**：「（見上文）」「（後詳）」這類含糊不要用
- **跨章**：明確章節號 + 簡短 1-2 字提示（如「詳細推進順序見 §3.6」）
- **附錄**：附錄項目要在主文有 cross-ref（否則為何寫附錄？）

## Skeleton 寫完前自查清單

在交給使用者前 cc 自查：

- [ ] 核心主張 1 句真的能單獨成立（不需要看完全文才懂）？
- [ ] 5 分鐘版讀者旅程的 TL;DR 真的 1 頁裝得下？
- [ ] Concept map home 數 ≤ 10？
- [ ] 章節 outline 每節「不回答什麼」欄都有寫？
- [ ] 結構性砍除清單列了 ≥ 5 處？
- [ ] 文末「使用者請審查」3-5 個關鍵問題已寫？
- [ ] 沒有用 TL;DR / FAQ / FYI 等英文縮寫當章節標題？

過了再給使用者。

## 預期 skeleton 迭代

skeleton 通常要 3-5 版：

| 版次 | 典型發現 |
|---|---|
| v1 | 章節順序、TL;DR 的 1 句不對 |
| v2 | concept map 有 1-2 個 home 該換家 |
| v3 | 某概念其實是另一個的屬性，要屬性化 |
| v4 | framework 本身切角錯，要重新分維（例：L1-L3 stack → L0-L3 risk） |
| v5+ | 細節調整 |

每版迭代後使用者反饋 → cc 寫 `_skeleton_v2.md` `_v3.md`...保留歷史。

**不要為了 v1 通過而過度妥協**。skeleton 是長文成敗關鍵，這階多花時間後面省更多。
